Please inspect and modify the current project code. Add a new “URL component position-aware n-gram vocabulary” feature to the existing URL n-gram vocabulary construction pipeline.

The project is for phishing URL detection. The current code already generates URL n-gram tokens, builds a vocabulary, and trains/evaluates a model. The new feature must be added without breaking the existing baseline behavior.

The main objective:

Original vocabulary tokens may look like:

    log
    ogi
    pay
    com
    id
    =1

The new position-aware vocabulary should generate tokens like:

    SUBDOMAIN::log
    DOMAIN::pay
    TLD::com
    PATH::aut
    QUERY_KEY::id
    QUERY_KEY+QUERY_VALUE::d=1
    PATH+QUERY_KEY::h?i
    DOMAIN+TLD::y.c

In other words, the same n-gram appearing in different URL components must be treated as different tokens. If an n-gram crosses the boundary between two or more URL components, the token prefix must be the merged component class.

==================================================
2. URL Component Classes
==================================================

Assign character-level component labels to the original URL using the following component classes:

    SCHEME
    SUBDOMAIN
    DOMAIN
    TLD
    PORT
    PATH
    QUERY_KEY
    QUERY_VALUE
    FRAGMENT

Example URL:

    https://login.pay.com:443/auth?id=12&tk=ab#top

Expected logical decomposition:

    SCHEME      = https
    SUBDOMAIN   = login
    DOMAIN      = pay
    TLD         = com
    PORT        = 443
    PATH        = auth
    QUERY_KEY   = id, tk
    QUERY_VALUE = 12, ab
    FRAGMENT    = top

Requirements:

1. Do not lowercase the URL.
2. Do not remove special characters.
3. Do not remove characters such as `.`, `/`, `?`, `=`, `&`, `#`, `:`, `-`, `_`.
4. Preserve case in path, query, and fragment.
5. Missing fields must be handled safely.
6. If precise public suffix parsing is not available, use a simple hostname split:
   - the last segment is TLD
   - the second-to-last segment is DOMAIN
   - preceding segments are SUBDOMAIN
7. If hostname has two segments, for example `pay.com`:
   - DOMAIN = pay
   - TLD = com
   - SUBDOMAIN is empty
8. If hostname has one segment, for example `localhost`:
   - DOMAIN = localhost
   - TLD is empty
   - SUBDOMAIN is empty
9. If host is an IP address, label the whole host as DOMAIN or HOST. Add a clear code comment explaining the chosen behavior.

==================================================
3. Boundary-Merged Component Class Rule
==================================================

This is the most important part of the modification.

Do not generate n-grams independently from each URL component, because that loses boundary information.

Correct approach:

1. Build character-level position labels for the full original URL.
2. Generate 1-gram, 2-gram, and 3-gram tokens by sliding over the full original URL.
3. For each n-gram, inspect the component labels covered by its characters.
4. If the n-gram belongs to only one component:

       token = COMPONENT::ngram

   Examples:

       DOMAIN::pay
       PATH::aut
       QUERY_KEY::id

5. If the n-gram crosses two or more components:

       token = COMPONENT_A+COMPONENT_B::ngram

   or:

       token = COMPONENT_A+COMPONENT_B+COMPONENT_C::ngram

   Examples:

       DOMAIN+TLD::y.c
       TLD+PORT::m:4
       PORT+PATH::3/a
       PATH+QUERY_KEY::h?i
       QUERY_KEY+QUERY_VALUE::d=1
       QUERY_VALUE+QUERY_KEY::2&t
       QUERY_VALUE+FRAGMENT::b#t

6. Component order in merged class must follow the order in which the components appear in the URL. Do not sort alphabetically.

For example:

    QUERY_VALUE+QUERY_KEY::2&t

must not become:

    QUERY_KEY+QUERY_VALUE::2&t

because the actual URL order is value to key.

==================================================
4. Separator Handling
==================================================

URL separators must be retained, including:

    ://
    .
    :
    /
    ?
    =
    &
    #

Recommended implementation:

1. Label normal characters with their URL component.
2. Label separator characters according to the components they connect.
3. When an n-gram window covers multiple labels, use the merged component class.

Examples:

For URL substring:

    pay.com

Approximate character labels:

    p a y . c o m
    DOMAIN DOMAIN DOMAIN DOMAIN+TLD TLD TLD TLD

Generated 3-grams:

    pay  -> DOMAIN::pay
    ay.  -> DOMAIN+TLD::ay.
    y.c  -> DOMAIN+TLD::y.c
    .co  -> DOMAIN+TLD::.co
    com  -> TLD::com

For URL substring:

    auth?id

Generated 3-grams:

    aut  -> PATH::aut
    uth  -> PATH::uth
    th?  -> PATH+QUERY_KEY::th?
    h?i  -> PATH+QUERY_KEY::h?i
    ?id  -> PATH+QUERY_KEY::?id

For URL substring:

    id=12

Generated 3-grams:

    id=  -> QUERY_KEY+QUERY_VALUE::id=
    d=1  -> QUERY_KEY+QUERY_VALUE::d=1
    =12  -> QUERY_KEY+QUERY_VALUE::=12

For URL substring:

    12&tk

Generated 3-grams:

    12&  -> QUERY_VALUE+QUERY_KEY::12&
    2&t  -> QUERY_VALUE+QUERY_KEY::2&t
    &tk  -> QUERY_VALUE+QUERY_KEY::&tk

For URL substring:

    ab#top

Generated 3-grams:

    ab#  -> QUERY_VALUE+FRAGMENT::ab#
    b#t  -> QUERY_VALUE+FRAGMENT::b#t
    #to  -> QUERY_VALUE+FRAGMENT::#to
    top  -> FRAGMENT::top
